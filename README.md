# High-Performance Product Catalog Backend

A production-quality Node.js, Express, and PostgreSQL backend managing a catalog of approximately 200,000 products. Built from scratch without ORMs to demonstrate raw SQL efficiency, optimized indexing, and robust cursor-based pagination.

---

## Technical Stack
- **Core**: Node.js & Express.js (ES Modules, async/await)
- **Database**: PostgreSQL (v16)
- **Driver**: `pg` (node-postgres connection pooling client)
- **Mock Data**: `@faker-js/faker`

## Setup & Installation

### Prerequisites
- Node.js (v18 or higher recommended)
- PostgreSQL (running locally or in a container)

### 1. Install Dependencies
```bash
npm install
```

### 2. Configure Environment Variables
Copy `.env.example` to `.env` and adjust database credentials:
```bash
cp .env.example .env
```
Ensure your connection URL matches your PostgreSQL credentials:
```ini
PORT=5000
NODE_ENV=development
DATABASE_URL=postgresql://postgres:YOUR_PASSWORD@localhost:5432/product_catalog
```

### 3. Bootstrap the Database & Schema
This script automatically connects to PostgreSQL, creates the target database if it doesn't exist (terminating template conflicts if needed), and applies all tables and indexes defined in `sql/schema.sql`:
```bash
npm run db:setup
```

### 4. Seed the Database (200,000 Products)
Executes a high-speed batch-insert seeding script:
```bash
npm run seed
```

### 5. Run the Server
For development (with automatic reloading via nodemon):
```bash
npm run dev
```
For production:
```bash
npm start
```

## API Documentation

### GET Products
Retrieves a page of products ordered by newest first (`created_at DESC, id DESC`).

**Endpoint:** `GET /api/products`

**Query Parameters:**
- `limit` (Optional): Integer between 1 and 100. Defaults to `20`.
- `category` (Optional): String matching one of the valid product categories: `electronics`, `fashion`, `books`, `sports`, `toys`, `home`, `beauty`, `automotive`.
- `cursor` (Optional): Base64 encoded JSON string generated by a previous page request.

#### Example Requests (cURL)

1. **Fetch first page (global):**
   ```bash
   curl -i "http://localhost:5000/api/products?limit=5"
   ```

2. **Fetch next page using cursor:**
   Copy the `nextCursor` value returned by the first request:

3. **Fetch first page filtered by category:**

## Design Decisions

### 1. Why PostgreSQL
PostgreSQL is a robust, enterprise-grade open-source relational database. For product catalogs of this size, it provides key benefits:
- Strong ACID compliance for product updates and inventory management.
- Highly optimized query execution planner with support for composite indexes.
- Low overhead connection pooling via Node.js, making it suitable for concurrent HTTP traffic.

### 2. Why Raw SQL
We consciously avoided ORMs (like Prisma or Sequelize) because:
- **Performance**: ORMs add abstraction layers, generate large/unoptimized queries, and carry object hydration overhead. Raw SQL keeps queries lean and fast.
- **Control**: Implementing keyset cursor pagination on ORMs can be verbose or unsupported natively. Raw SQL allows us to define the precise multi-column comparison (`(created_at, id) < ($1, $2)`) and ensures the index is scanned correctly.
- **Transparency**: Every SQL statement is explicitly written, ensuring clear visibility into what is run on the server.

### 3. Why Cursor Pagination
In large datasets, **Offset Pagination** (`LIMIT 20 OFFSET 100000`) becomes unusable:
- **Performance degradation (O(N) search)**: PostgreSQL must read, sort, and discard all N records before return limits. At high offsets, queries degrade from milliseconds to seconds.
- **Data drift (Duplicates & Missing records)**:
  - If a product is **inserted** while a user is browsing, rows shift down. The user sees the last record of Page 1 repeat on the top of Page 2.
  - If a product is **deleted** while browsing, rows shift up. The user misses a record when going to Page 2.
- **How Cursor Pagination Solves This**:
  By specifying the exact record coordinate (`created_at < cursor.createdAt OR (created_at = cursor.createdAt AND id < cursor.id)`), queries jump directly to the target record via index range scan (O(log N)). The offset size is irrelevant, maintaining constant execution time. Since the boundary is anchored to unique record properties rather than record counts, background database updates never cause row duplication or skipping.

### 4. Why Composite Indexes
We defined two specialized composite indexes in `schema.sql`:
1. `idx_products_created_id` on `(created_at DESC, id DESC)`
2. `idx_products_category_created_id` on `(category, created_at DESC, id DESC)`

- **Why DESC?** Our search returns the newest items first. Indexing in matching order allows the query planner to read the index forward, eliminating the need for sorting (zero filesort overhead).
- **Why include `id`?** Multiple items can be created at the exact same timestamp. Adding the primary key `id` guarantees uniqueness and enforces a stable, predictable cursor ordering.
- **Why category first?** By placing `category` at the front of the second index, the query engine performs an equality lookup on category first, then instantly retrieves the remaining matching items pre-sorted by date and ID.

### 5. Why Batch Inserts
Running individual `INSERT` queries for 200,000 products would cause severe database network round-trip bottlenecks (taking minutes to complete). 
Our seeding script generates multi-row insert statements:
```sql
INSERT INTO products (...) VALUES (...), (...), ...;
```
Using a batch size of 5,000 inserts 30,000 parameters per query (well below the PostgreSQL uint16 limit of 65,535 parameters). This cuts database round-trips from 200,000 to just 40, allowing the entire 200,000 product insertion to finish in under 10 seconds.

## Scalability to 1 Million / 10 Million Products

To scale this backend system past 10 million products, we would apply the following strategies:

### 1. Read Replicas
Product catalogs are read-heavy (95%+ reads). By setting up a primary-replica cluster:
- **Primary Node**: Handles write queries (inserting and updating products).
- **Replica Nodes**: Handle all paginated `GET` requests.
- Node.js database pools can be configured to direct all catalog queries to replicas, scaling query throughput horizontally.

### 2. Caching (Redis)
Popular pages (e.g. the first few pages of products or common category queries) can be cached using a Redis cluster.
- Cursors serve as deterministic cache keys (e.g. `products:category:electronics:limit:20:cursor:xyz`).
- Write-through or write-behind cache invalidation clears hot keys when new items are added to a category.
- Drastically reduces database CPU usage under spike traffic.

### 3. Database Partitioning
At 10 million rows, the index size grows and may exceed RAM capacity.
- **Range Partitioning by Time**: We can partition the `products` table by month/quarter based on `created_at`. Since query traffic is heavily biased toward "newest first", queries will target only the latest partitions, allowing older partitions to be stored on slower disks and keeping the active index footprint small.
- **Hash Partitioning by Category**: If category filtering is the main access pattern, partitioning on category routes queries to smaller, isolated partition tables.

### 4. Search Services (Elasticsearch / Meilisearch)
Relational databases are not designed for full-text fuzzy search or complex multi-faceted filtering.
- Sync product changes to a dedicated search cluster (like Elasticsearch or Meilisearch) using a Change Data Capture (CDC) pipeline.
- Offload catalog search and category filtering to the search service, keeping PostgreSQL isolated for transactional operations.
